    Checklist – Accès à la production Google Play (Yin Shi)

    Cette checklist a vocation à :

    • Guider l’équipe pour cette campagne de test fermé.
    • Servir de base pour les futures applications soumises avec un compte développeur personnel.

    1. Avant de (re)lancer la campagne de test fermé

    • [x] Objectifs clarifiés
    • [x] Toute l’équipe comprend le problème (“More testing required…”) et les exigences Google.

    • [x] La vision et la Definition of Done globale sont validées (voir GOOGLE_PLAY_PRODUCTION_STRATEGIE_AGILE.md).

    • [x] État initial documenté

    • [x] Nombre actuel de testeurs, pays, date de début du test fermé notés.

    • [x] Dernière release de test fermé (ex : 1.0.20) et son contenu sont connus.

    • [x] Fenêtre temporelle décidée

    • [x] Date de l’examen Google : 19/11/2025, 21:04.
    • [x] Fenêtre de test cible : du 20/11/2025 au 10/12/2025 (≈ 3 semaines incluant les 14 jours requis).

    2. Recrutement et organisation des testeurs

    • [x] Cible de testeurs définie
    • [x] Profil type des testeurs (pays, type d’appareil, usage cible) clarifié.

    • [x] Volume cible de testeurs fixé (par ex. ≥ 20 pour sécuriser le seuil des 12).

    • [ ] Canaux de recrutement prêts

    • [ ] Liste des canaux (amis, collègues, communautés, réseaux sociaux) définie.

    • [x] Message type d’invitation rédigé (FR/EN si besoin), incluant :

      • [x] Lien de participation au test fermé.
      • [x] Explication de la durée de participation (≥ 14 jours consécutifs sans se désinscrire).
      • [x] Description succincte de l’app et de ce qu’on attend d’eux.

    • [ ] Guide testeurs disponible

    • [x] Document simple décrivant :
      • [x] Les scénarios principaux à tester (onboarding, fonctionnalités clés, etc.).
      • [x] Comment signaler les bugs ou problèmes d’UX.
    • [x] Utiliser comme modèle le GOOGLE_PLAY_TESTERS_GUIDE.md.

    3. Mise en place du feedback

    • [ ] Canal principal de feedback choisi
    • [ ] Google Form / groupe Discord / WhatsApp / email dédié (un ou plusieurs).

    • [ ] Template de feedback créé (titre, description, device, version, captures, catégorie, gravité).

    • [ ] Communication aux testeurs

    • [ ] Les testeurs savent comment envoyer du feedback.

    • [ ] Les testeurs savent qu’ils peuvent aussi utiliser le feedback privé de test dans Google Play.

    • [ ] Organisation interne

    • [ ] Responsable désigné pour lire / trier régulièrement le feedback.
    • [ ] Décision sur la fréquence de tri (ex : 1–2 fois par semaine).

    4. Pendant les 14+ jours de test fermé

    • [ ] Nombre de testeurs
    • [ ] Vérifier régulièrement que ≥ 12 testeurs sont opt‑in en continu.

    • [ ] Relancer quelques testeurs si besoin pour maintenir le seuil.

    • [ ] Engagement

    • [ ] Encourager les testeurs à utiliser l’app plusieurs fois pendant la période.

    • [ ] S’assurer que les features clés sont bien testées (rappels, messages, etc.).

    • [ ] Releases d’amélioration

    • [ ] Planifier au moins 1–2 nouvelles versions sur la piste de test fermé.

    • [ ] Pour chaque release :

      • [ ] Scope défini (bugs/UX à corriger en priorité).
      • [ ] Changelog simple rédigé pour les testeurs.
      • [ ] Communication envoyée aux testeurs pour leur signaler la nouvelle version.

    • [ ] Suivi en Console

    • [ ] Suivre les éventuels crashs/ANR (si l’outillage est en place).
    • [ ] Lire et, idéalement, répondre aux “Testing feedback” dans la Play Console.

    5. Avant la nouvelle demande d’accès à la production

    • [ ] Validation des critères Google
    • [ ] Les 14 jours consécutifs après la date d’examen sont écoulés.
    • [ ] Durant ces 14 jours, il y a eu ≥ 12 testeurs opt‑in en continu.

    • [ ] Au moins 1–2 releases ont été publiées sur la piste de test fermé pendant cette période.

    • [ ] Synthèse de la campagne prête

    • [ ] Timeline des tests (dates, releases, événements) rédigée.

    • [ ] Principaux problèmes remontés et solutions/apports listés.

    • [ ] Pré‑rédaction du formulaire “Production access”

    • [ ] Section “About your closed test” :
      • [ ] Comment les testeurs ont été recrutés.
      • [ ] Comment ils ont utilisé l’app (scénarios, fréquence, devices).
      • [ ] Comment le feedback a été collecté.
      • [ ] Résumé du feedback + ce qui a été corrigé.
    • [ ] Section “About your app/game” :
      • [ ] Audience cible.
      • [ ] Valeur apportée aux utilisateurs.
      • [ ] Estimation du volume d’installations la première année.
    • [ ] Section “Production readiness” :
      • [ ] Changements effectués suite à la campagne de tests.
      • [ ] Raisons pour lesquelles l’app est considérée prête (stabilité, feedback positif, flows critiques validés).

    6. Après soumission

    • [ ] Suivi de la revue Google
    • [ ] Email de confirmation de la soumission reçu.

    • [ ] Délai indicatif compris (souvent ≤ 7 jours).

    • [ ] Résultat

    • [ ] Si acceptée :
      • [ ] Accès à la Production confirmé.
      • [ ] Publication de l’app planifiée (open testing / production directe selon stratégie).
    • [ ] Si refusée :
      • [ ] Analyser précisément les nouveaux motifs.
      • [ ] Mettre à jour ce dossier (rapport / stratégie / checklist) en conséquence.

    7. Capitalisation

    • [ ] Documentation mise à jour
    • [ ] GOOGLE_PLAY_PRODUCTION_RAPPORT.md reflète l’historique complet.

    • [ ] GOOGLE_PLAY_PRODUCTION_STRATEGIE_AGILE.md mis à jour si nécessaire.

    • [ ] Rétrospective réalisée

    • [ ] Ce qui a bien/mal fonctionné pendant cette campagne est discuté.
    • [ ] 2–3 actions d’amélioration identifiées pour les prochains lancements.

    Cette checklist est volontairement détaillée : l’équipe peut choisir de cocher tout, ou de définir un “noyau minimum” pour des cas plus simples, tant que les exigences Google sont respectées.
